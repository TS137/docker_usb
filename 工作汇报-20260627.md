## 一、本周工作概述

本周完成了基于 Docker 的本地 USB 摄像头 RTMP 推流服务器的基础服务搭建，实现了从摄像头采集到 Web 端实时预览的完整链路。方案采用宿主机 Arch Linux + 容器 Ubuntu 的架构，使用 Docker Compose 编排两个服务容器，支持一键部署。

## 二、整体架构设计

系统由两个 Docker 容器组成，通过 Docker Compose 桥接网络通信：

```
┌─────────────────────────────────────────────────────────┐
│                    Docker Compose Network                │
│                                                         │
│  ┌──────────────────────┐    ┌──────────────────────┐   │
│  │   camera-streamer    │    │     nginx-rtmp       │   │
│  │                      │    │                      │   │
│  │  ubuntu:24.04        │    │  tiangolo/nginx-rtmp │   │
│  │  ├─ ffmpeg (V4L2)    │    │  ├─ RTMP ingest      │   │
│  │  ├─ libx264 编码     │RTMP│  ├─ HLS 转封装       │   │
│  │  └─ tee muxer 双输出 │───>│  └─ Web 静态服务     │   │
│  │                      │:1935│                      │   │
│  │  挂载 /dev/video0    │    │  bind mount ./hls    │   │
│  └──────────────────────┘    └──────────┬───────────┘   │
│                                         │               │
│                                    HTTP :8080            │
│                                    HLS (.m3u8/.ts)      │
└─────────────────────────────────────────│───────────────┘
                                          │
                                          ▼
                                   ┌────────────┐
                                   │   浏览器    │
                                   │  hls.js    │
                                   │  播放器    │
                                   └────────────┘
```

**数据流路径**: USB摄像头 → V4L2 采集(YUV) → libx264 编码(H.264) → RTMP 推流 → Nginx RTMP 接收 → HLS 转封装 → 浏览器 hls.js 播放

## 三、详细实现过程

### 3.1 项目初始化

创建项目仓库并搭建基础文件结构，包含服务编排、镜像构建、配置管理等核心文件：

```
docker_usb/
├── docker-compose.yml      # 两容器服务编排
├── .dockerignore           # 构建上下文过滤
├── .env.example            # 环境变量模板
├── Dockerfile.camera       # 采集端镜像 (ubuntu:24.04 + ffmpeg)
├── Dockerfile.nginx        # 服务端镜像 (tiangolo/nginx-rtmp)
├── nginx.conf              # RTMP 接收 + HLS 分发配置
├── entrypoint.sh           # 采集容器入口脚本
├── setup.sh                # 交互式一键部署脚本
├── web/index.html          # Web 预览页面 (hls.js)
└── README.md
```

### 3.2 采集容器实现 (Dockerfile.camera)

基于 `ubuntu:24.04` 构建，通过 apt 安装 ffmpeg 和 netcat：

```dockerfile
FROM ubuntu:24.04

ENV DEBIAN_FRONTEND=noninteractive

RUN apt-get update && \
    apt-get install -y --no-install-recommends ffmpeg netcat-openbsd && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

RUN mkdir -p /app
COPY entrypoint.sh /app/entrypoint.sh
RUN chmod +x /app/entrypoint.sh

ENTRYPOINT ["/app/entrypoint.sh"]
```

### 3.3 编码推流实现 (entrypoint.sh)

核心采集逻辑 -- 使用 ffmpeg 从 V4L2 接口采集 MJPEG 格式画面，libx264 软件编码，RTMP 推送：

```bash
# 关键帧间隔计算：GOP = 帧率 × 2，确保每 2 秒一个关键帧
# 配合 HLS 3秒分片，保证每个分片都有关键帧
GOP=$((FRAMERATE * 2))

ffmpeg \
    -f v4l2 -input_format mjpeg \
    -video_size "${RESOLUTION}" -framerate "${FRAMERATE}" \
    -i "${CAMERA_DEVICE}" \
    -c:v libx264 -preset veryfast -tune zerolatency \
    -g "${GOP}" -keyint_min "${GOP}" -sc_threshold 0 \
    -b:v "${BITRATE}" -an \
    -f flv -flvflags no_duration_filesize \
    "${RTMP_TARGET}"
```

外部双路推流时使用 ffmpeg `tee` muxer 单次采集同步输出两个 RTMP 流，避免重复打开 `/dev/video0` 导致设备锁冲突。

### 3.4 RTMP 服务器实现 (nginx.conf)

Nginx 配置 RTMP 模块接收推流，并实时转封装为 HLS 分片：

```nginx
rtmp {
    server {
        listen 1935;
        chunk_size 4096;

        application live {
            live on;
            record off;

            hls on;
            hls_path /tmp/hls;
            hls_fragment 3s;        # 3秒一个分片
            hls_playlist_length 30s; # 保留30秒窗口
            hls_cleanup on;          # 自动清理过期分片
        }
    }
}
```

### 3.5 Web 预览页面 (web/index.html)

使用 hls.js 库实现浏览器端 HLS 播放，实时显示推流状态：

- HLS 流地址: `/hls/stream.m3u8`
- 播放器状态指示: 直播中 (Live) / 重连中 (Reconnecting) / 已断开 (Offline)
- 适配移动端响应式布局

交互式检测摄像头设备、选择分辨率和码率，自动生成 `.env` 配置并启动服务：

## 五、问题排查与修复

本周开发过程中发现并修复了以下关键问题：

| 问题 | 根因 | 修复方案 |
|------|------|----------|
| 双 ffmpeg 进程竞争 `/dev/video0` | V4L2 设备独占锁，第二个进程打开失败 | 改用 `tee` muxer 单次采集双路输出 |
| nginx-mod-rtmp 版本不兼容 | nginx:alpine 已升级到 1.31.2，Alpine 仓库仅有 1.28.3 的 RTMP 模块 | 切换为 `tiangolo/nginx-rtmp` 镜像 |
| HLS 播放列表仅 1 个分片 | libx264 默认 GOP=250 帧(30fps 下约 8.3s)，分片器按关键帧对齐导致异常 | 显式设置 `-g 60`(2秒一个关键帧) |
| `.env` 文件传入构建上下文 | 未配置 `.dockerignore`，Docker 尝试打包 `.env` 导致 tar 错误 | 添加 `.dockerignore` 排除 `.env` 和 `hls/` |
| docker-compose version 废弃警告 | 新版 Docker Compose 不再需要 `version` 字段 | 移除 `version: "3.8"` |

## 六、技术方案总结

本方案的核心优势：

1. **零硬件依赖**: 纯 libx264 软件编码，不依赖 GPU 驱动，兼容 Intel/AMD/NVIDIA 全平台
2. **一键部署**: `./setup.sh` 交互式配置，自动检测摄像头、生成参数并启动服务
3. **容器化隔离**: 宿主机仅需 Docker，采集环境完全封装在 Ubuntu 容器内，不受 Arch Linux 滚动更新影响
4. **灵活推流**: 内置 RTMP 服务端用于局域网预览，同时支持推流到外部平台（B站、YouTube 等）
5. **低延迟优化**: GOP=2s + HLS=3s，端到端延迟控制在 5-8 秒
